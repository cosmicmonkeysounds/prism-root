# @prism/studio

The Universal Host — Vite SPA + Tauri 2.0 desktop shell + Capacitor mobile wrapper.

## Build
- `pnpm dev` — Vite dev server on :1420
- `pnpm build` — production build
- `pnpm tauri dev` — Tauri desktop with hot reload
- `pnpm typecheck`
- `pnpm exec cap sync ios|android` — sync built `dist/` into native shells
- `pnpm exec cap add ios|android` — one-time scaffold of native projects (not checked in)

## Architecture
- Vite SPA (NOT Next.js) — local-first philosophy
- Tauri 2.0 shell wraps the SPA for desktop
- Capacitor 7 wraps the same SPA for iOS/Android (`capacitor.config.ts`)
- All daemon communication via `src/ipc-bridge.ts` using Tauri invoke()
- Never raw HTTP between frontend and daemon

## Self-replication (App Builder)
Studio composes focused apps (Flux / Lattice / Cadence / Grip / Relay) and
produces web / Tauri / Capacitor builds from the same codebase. Pipeline:
1. `BuilderManager` (`@prism/core/builder`) composes a `BuildPlan` from an
   `AppProfile` + `BuildTarget`. Studio just consumes it.
2. Each step is dispatched to the Prism Daemon via Tauri IPC
   (`invoke("run_build_step", { step, workingDir, env })`).
3. The daemon (`packages/prism-daemon/src/commands/build.rs`) resolves
   paths against `workingDir`, executes the step (emit-file, run-command,
   or invoke-ipc), and returns captured stdout/stderr.
4. For mobile targets, native scaffolds under `ios/` and `android/` are
   generated by running `pnpm cap add ios|android` once per checkout.
   Those directories are gitignored.

## Kernel (`src/kernel/`)
The Studio kernel wires all `@prism/core` subsystems together. All support modules (relay manager, builder manager, design tokens, page export, block style) were extracted to `@prism/core` in ADR-002 Phase 3 — Studio only keeps wiring + entity definitions here.
- `studio-kernel.ts` — `createStudioKernel({ lensBundles?, initializers? })`. Creates and connects ObjectRegistry, CollectionStore, PrismBus, AtomStore, ObjectAtomStore, UndoRedoManager, NotificationStore, SearchEngine, ActivityStore, ActivityTracker, RelayManager (from `@prism/core/relay-manager`), BuilderManager (from `@prism/core/builder`), AutomationEngine, graph analysis, expression evaluator, PluginRegistry, InputRouter, VaultRoster, FormState helpers, Identity (DID generation/sign/verify/export/import), VfsManager (content-addressed blob storage with locks), Trust & Safety (PeerTrustGraph, SchemaValidator, LuauSandbox, ShamirSplitter, EscrowManager), Facet System (FacetParser, SpellChecker, ProseCodec, Sequencer, Emitters, FacetDefinition registry), DesignTokenRegistry (from `@prism/core/design-tokens`). Also wires clipboard (copy/cut/paste), batch ops, templates, LiveView, **and the lens system**: the kernel owns its own `LensRegistry`, `lensComponents: Map<LensId, ComponentType>`, and `shellStore` (zustand). Callers pass `lensBundles` (typically `createBuiltinLensBundles()`) and each bundle self-registers its manifest + React component. `App.tsx` no longer touches lens plumbing — it just reads `kernel.lensRegistry`, `kernel.lensComponents`, `kernel.shellStore`. `dispose()` tears down every installed bundle + initializer in reverse order.
- `initializer.ts` — thin Studio-specialized alias over `@prism/core/initializer`. Exports `StudioInitializer = KernelInitializer<StudioKernel>` + re-exports `installInitializers` from Core. The generic `KernelInitializer<TKernel>` pattern lives in Core so Flux/Lattice/Musica can specialise it with their own kernel types.
- `builtin-initializers.ts` — the canonical `createBuiltinInitializers()` list: `pageTemplatesInitializer` (blog + landing page templates), `sectionTemplatesInitializer` (hero / CTA / pricing / footer), `demoWorkspaceInitializer` (seeds Home + About pages into an empty store, clears undo history, selects the home page), `dynamicDataInitializer` (seeds ~25 sample dynamic records — tasks, reminders, contacts, events, notes, goals, habits, bookmarks, captures — guarded on an empty store for those record types so it never clobbers user data).
- `entities.ts` — page-builder entity types (folder, page, section, heading, text-block, image, button, card, luau-block, facet-view, spatial-canvas, data-portal) + **record types** (task, reminder, contact, event, note, goal, habit, bookmark, timer-session, capture) + **dynamic widgets** (tasks-widget, reminders-widget, contacts-widget, events-widget, notes-widget, goals-widget, habit-tracker-widget, bookmarks-widget, timer-widget, capture-inbox-widget) with category containment rules and edge types (references, links-to). The `record` category (`canParent: []`, `canBeRoot: true`) hosts free-standing data rows at the workspace root; dynamic widgets are `component` (`childOnly: true`) that read records via `kernel.store.allObjects()`. Imports `STYLE_FIELD_DEFS` from `@prism/core/page-builder`.
- `kernel-context.tsx` — React context + hooks: useKernel, useSelection, useObjects, useObject, useUndo, useNotifications, useRelay, useConfig, useConfigSettings, usePresence, useAutomation, useGraphAnalysis, useExpression, usePlugins, useInputRouter, useVaultRoster, useIdentity, useVfs, useTrust, useFacetParser, useSpellCheck, useProseCodec, useSequencer, useEmitters, useFacetDefinitions
- `use-registration.ts` — canonical "validate → exists? → register → notify → reset" flow for builder panels (entity-builder / relationship-builder / schema-designer). Exports the React `useRegistration<TDef>({ noun, name, validate, exists, register, onSuccess })` hook + a pure `buildRegistration(kernel, options)` factory so tests can drive the flow against a stub `RegistrationKernel` (`{ notifications: { add } }`) without mounting React. Returns `(def) => boolean` — `false` when validation or the uniqueness check blocked the write, so callers can gate follow-up side effects (e.g. `setEdges` in the schema designer's `onConnect`).

## Lenses
Every panel self-registers via a `LensBundle` colocated with its component. A panel file (e.g. `panels/editor-panel.tsx`) exports both its React component and a `xxxLensBundle = defineLensBundle(manifest, Component)` created with `defineLensBundle` from `src/lenses/bundle.ts` (Studio's `ComponentType`-specialized re-export of `@prism/core/lens`'s generic `defineLensBundle`). `src/lenses/index.tsx` is a pure auto-aggregator: it uses Vite's `import.meta.glob<..., { eager: true }>("../panels/*-panel.tsx")` at bundle time and hands the module map to `buildLensBundleList` from `./collect.ts`, which scans every `*LensBundle` export that matches the shape check (`typeof id === "string" && typeof install === "function"`), sorts by file path for deterministic ordering, dedupes by id, and returns the canonical list. `createBuiltinLensBundles()` is the public entry point. The kernel installs them via `installLensBundles` inside `createStudioKernel`, so manifests + components land in `kernel.lensRegistry` + `kernel.lensComponents` atomically — no parallel registries, no manifest list to sync.

Pure scanning helpers live in `src/lenses/collect.ts` (`isLensBundle`, `collectLensBundlesFromModule`, `buildLensBundleList`). Tests import from `./collect.js`, not `./index.js`, so the glob never fires during vitest runs (it would otherwise eager-load every panel including browser-only transitive deps like leaflet/window).

Adding a new lens:
1. Create `panels/my-panel.tsx` exporting `MyPanel` and `myLensBundle`.
2. That's it — the auto-glob picks it up the next time Vite rebuilds.

## Shell Mode & Permission
Studio boots into a `(shellMode, permission)` context picked at launch:
- **`shellMode`** — `use` / `build` / `admin`. Each owns its own Puck shell tree (`USE_SHELL_TREE` / `BUILD_SHELL_TREE` / `ADMIN_SHELL_TREE` in `@prism/core/puck`) and its own visible lens set. Mutable at runtime via `kernel.setShellMode(mode)` or `Cmd+Shift+E`. Switching notifies both `onShellModeChange` and `onShellTreeChange`.
- **`permission`** — `user` / `dev`. Frozen at boot (no setter). Gates both which lenses are reachable (`minPermission`) and which daemon commands the IPC bridge can invoke.

Each `LensBundle` declares optional `availableInModes` + `minPermission` via `withShellModes(bundle, { ... })`. Defaults: `availableInModes = ["build", "admin"]` and `minPermission = "user"` — a bundle with no constraints is hidden in `use` but reachable by any tier in `build`/`admin`. `kernel.getVisibleLensIds()` / `kernel.isLensVisible(id)` filter by the current context; `filterLensBundlesByShellMode(bundles, ctx)` is the pure helper if you need to drive the filter outside the kernel.

`loadBootConfig` in `src/boot/load-boot-config.ts` resolves the runtime `BootConfig` from four sources in precedence order: query params → `VITE_PRISM_BOOT_CONFIG` env var → build-time `VITE_PRISM_BOOT_DEFAULT` ceiling → `DEFAULT_BOOT_CONFIG`. A URL that tries to escalate permission past the ceiling is clamped with a console warning.

The `prism-studio` Node bin (`bin/prism-studio.mjs`) translates subcommands to boot configs: `run` → `use`/`user`, `build` → `build`/`dev`, `admin` → `admin`/`dev`, `dev` / `bundle` → no override. `--profile=<id>` folds into the boot config so focused-app profiles (flux / lattice / musica / …) pin the lens set. Pure helpers (`buildBootConfigForSubcommand`, `extractProfileFlag`, `stripProfileFlag`, `buildChildEnv`, `viteArgsForSubcommand`) are exported for vitest — the test file lives at `src/bin/prism-studio-launcher.test.ts` because the root vitest glob is `packages/*/src/**/*.test.{ts,tsx}`.

See `docs/dev/panel-modes.md` for the full user-facing guide.

## Data Flow
```
User action → kernel.createObject/updateObject/deleteObject
  → CollectionStore.putObject (Loro CRDT)
  → PrismBus.emit (ObjectCreated/Updated/Deleted)
  → connectBusToObjectAtoms → ObjectAtomStore (Zustand cache)
  → React components re-render via useSyncExternalStore
  → UndoRedoManager.push (snapshot for undo)
  → SearchEngine auto-reindexes via CollectionStore.onChange
```

## Key Design Decisions
- Kernel is a singleton created at app startup, not per-component
- All mutations go through kernel CRUD methods (never raw CollectionStore)
- Bus payloads follow connectBusToObjectAtoms format: `{ object }`, `{ edge }`, `{ id }`
- StudioShell replaces core ShellLayout to inject real sidebar/inspector content
- Panels access data via kernel context (useKernel), not prop drilling
- Editor creates per-object LoroText buffers keyed as `obj_content_{id}`, debounce-syncs back to kernel
- Canvas resolves the selected page (walking up parentId if a child is selected) and renders live
- Studio has NO server code — Relay servers are managed via CLI, Studio connects as a client
- RelayManager uses HTTP for portal CRUD, WebSocket (RelayClient SDK) for live CRDT sync
