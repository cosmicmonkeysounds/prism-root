# @prism/studio

The Universal Host — Vite SPA + Tauri 2.0 desktop shell + Capacitor mobile wrapper.

## Build
- `pnpm dev` — Vite dev server on :1420
- `pnpm build` — production build
- `pnpm tauri dev` — Tauri desktop with hot reload
- `pnpm typecheck`
- `pnpm exec cap sync ios|android` — sync built `dist/` into native shells
- `pnpm exec cap add ios|android` — one-time scaffold of native projects (not checked in)

## Architecture
- Vite SPA (NOT Next.js) — local-first philosophy
- Tauri 2.0 shell wraps the SPA for desktop
- Capacitor 7 wraps the same SPA for iOS/Android (`capacitor.config.ts`)
- All daemon communication via `src/ipc-bridge.ts` using Tauri invoke()
- Never raw HTTP between frontend and daemon

## Self-replication (App Builder)
Studio composes focused apps (Flux / Lattice / Cadence / Grip / Relay) and
produces web / Tauri / Capacitor builds from the same codebase. Pipeline:
1. `BuilderManager` (`src/kernel/builder-manager.ts`) composes a `BuildPlan`
   from an `AppProfile` + `BuildTarget` via `@prism/core/builder`.
2. Each step is dispatched to the Prism Daemon via Tauri IPC
   (`invoke("run_build_step", { step, workingDir, env })`).
3. The daemon (`packages/prism-daemon/src/commands/build.rs`) resolves
   paths against `workingDir`, executes the step (emit-file, run-command,
   or invoke-ipc), and returns captured stdout/stderr.
4. For mobile targets, native scaffolds under `ios/` and `android/` are
   generated by running `pnpm cap add ios|android` once per checkout.
   Those directories are gitignored.

## Kernel (`src/kernel/`)
The Studio kernel wires all `@prism/core` subsystems together:
- `studio-kernel.ts` — `createStudioKernel({ lensBundles?, initializers? })`. Creates and connects ObjectRegistry, CollectionStore, PrismBus, AtomStore, ObjectAtomStore, UndoRedoManager, NotificationStore, SearchEngine, ActivityStore, ActivityTracker, RelayManager, AutomationEngine, graph analysis, expression evaluator, PluginRegistry, InputRouter, VaultRoster, FormState helpers, Identity (DID generation/sign/verify/export/import), VfsManager (content-addressed blob storage with locks), Trust & Safety (PeerTrustGraph, SchemaValidator, LuauSandbox, ShamirSplitter, EscrowManager), Facet System (FacetParser, SpellChecker, ProseCodec, Sequencer, Emitters, FacetDefinition registry). Also wires clipboard (copy/cut/paste), batch ops, templates, LiveView, **and the lens system**: the kernel owns its own `LensRegistry`, `lensComponents: Map<LensId, ComponentType>`, and `shellStore` (zustand). Callers pass `lensBundles` (typically `createBuiltinLensBundles()`) and each bundle self-registers its manifest + React component. `App.tsx` no longer touches lens plumbing — it just reads `kernel.lensRegistry`, `kernel.lensComponents`, `kernel.shellStore`. `dispose()` tears down every installed bundle + initializer in reverse order.
- `initializer.ts` — `StudioInitializer` interface + `installInitializers()`. An initializer is any `{ id, name, install({ kernel }) => uninstall }` hook that runs AFTER the kernel's return object exists, so it can freely call `kernel.registerTemplate`, `kernel.createObject`, etc. Symmetric with `PluginBundle` / `LensBundle` but scoped to post-boot side effects.
- `builtin-initializers.ts` — the canonical `createBuiltinInitializers()` list: `pageTemplatesInitializer` (blog + landing page templates), `sectionTemplatesInitializer` (hero / CTA / pricing / footer), `demoWorkspaceInitializer` (seeds Home + About pages into an empty store, clears undo history, selects the home page). Replaces the old ad-hoc `registerSeedTemplates()` / `seedDemoData()` helpers that used to live at the top of `App.tsx`.
- `relay-manager.ts` — manages connections to Prism Relay servers. Studio is client-only (no server code). Handles relay CRUD, WebSocket connect/disconnect via RelayClient SDK, portal publish/unpublish/list via HTTP API, collection sync, and relay status. Injectable HTTP/WS clients for testing.
- `entities.ts` — page-builder entity types (folder, page, section, heading, text-block, image, button, card, luau-block, facet-view, spatial-canvas, data-portal) with category containment rules and edge types (references, links-to)
- `kernel-context.tsx` — React context + hooks: useKernel, useSelection, useObjects, useObject, useUndo, useNotifications, useRelay, useConfig, useConfigSettings, usePresence, useAutomation, useGraphAnalysis, useExpression, usePlugins, useInputRouter, useVaultRoster, useIdentity, useVfs, useTrust, useFacetParser, useSpellCheck, useProseCodec, useSequencer, useEmitters, useFacetDefinitions

## Kernel helpers (`src/kernel/*`)
- `block-style.ts` — shared `BlockStyleData` shape, `STYLE_FIELD_DEFS` (spread into section/heading/text-block/button/card entity defs), `computeBlockStyle()`/`extractBlockStyle()`/`mergeCss()`/`resolveShadow()` — every renderer and the HTML exporter apply the same per-block styling.

## Lenses
Every panel self-registers via a `LensBundle` colocated with its component. A panel file (e.g. `panels/editor-panel.tsx`) exports both its React component and a `xxxLensBundle = defineLensBundle(manifest, Component)` created with `defineLensBundle` from `src/lenses/bundle.ts` (Studio's `ComponentType`-specialized re-export of `@prism/core/lens`'s generic `defineLensBundle`). `src/lenses/index.tsx` is a pure aggregator: it imports each bundle and exposes `createBuiltinLensBundles()`. The kernel installs them via `installLensBundles` inside `createStudioKernel`, so manifests + components land in `kernel.lensRegistry` + `kernel.lensComponents` atomically — no parallel registries, no manifest list to sync.

Adding a new lens:
1. Create `panels/my-panel.tsx` exporting `MyPanel` and `myLensBundle`.
2. Add `myLensBundle` to the list in `lenses/index.tsx`.

## Data Flow
```
User action → kernel.createObject/updateObject/deleteObject
  → CollectionStore.putObject (Loro CRDT)
  → PrismBus.emit (ObjectCreated/Updated/Deleted)
  → connectBusToObjectAtoms → ObjectAtomStore (Zustand cache)
  → React components re-render via useSyncExternalStore
  → UndoRedoManager.push (snapshot for undo)
  → SearchEngine auto-reindexes via CollectionStore.onChange
```

## Key Design Decisions
- Kernel is a singleton created at app startup, not per-component
- All mutations go through kernel CRUD methods (never raw CollectionStore)
- Bus payloads follow connectBusToObjectAtoms format: `{ object }`, `{ edge }`, `{ id }`
- StudioShell replaces core ShellLayout to inject real sidebar/inspector content
- Panels access data via kernel context (useKernel), not prop drilling
- Editor creates per-object LoroText buffers keyed as `obj_content_{id}`, debounce-syncs back to kernel
- Canvas resolves the selected page (walking up parentId if a child is selected) and renders live
- Studio has NO server code — Relay servers are managed via CLI, Studio connects as a client
- RelayManager uses HTTP for portal CRUD, WebSocket (RelayClient SDK) for live CRDT sync
